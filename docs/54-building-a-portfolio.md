# Chapter 54: Building a Portfolio

If your goal is to get a job in software engineering, tech companies do not care about certificates or what tutorials you have watched. They care about one thing: **Can you build working software?**

The only way to prove this is by building a Portfolio on GitHub. Now that you have a professional Linux workstation, you are perfectly equipped to start writing code that matters.

## Learning Objectives
By the end of this chapter, you will:
- Understand the importance of a GitHub profile.
- Learn what makes a good portfolio project.
- Learn how to contribute to Open Source.

---

## Theory: The GitHub Resume

In the tech industry, your GitHub profile is your resume. When you apply for a job, the hiring manager will click your GitHub link and look at three things:
1. **The Green Graph:** Are you coding consistently? (A graph full of green squares shows dedication).
2. **Your Repositories:** What kind of projects are you building? Are they complex?
3. **Your Code Quality:** When they open a project, is the code formatted well? Is there a good `README.md` explaining what the project does?

### What to Build
Beginners often build "To-Do Apps" or "Weather Apps" by following YouTube tutorials. Hiring managers see thousands of these. They ignore them.

You need to build projects that solve **real problems**, even if they are small.
- Did you write a Python script (Chapter 31) that scrapes real estate data for your local town?
- Did you build a Node.js API (Chapter 30) that serves random facts about your favorite video game?
- Did you create a React frontend hosted on Vercel (Chapter 38) that consumes that API?

These projects prove you can think independently, design architecture, and deploy to the real world.

## Practical Example: Open Source Contribution

You don't just have to build your own projects. You can contribute to existing, massive projects. This is called **Open Source**.

Because you are running Ubuntu (an open-source OS) and using Git (Chapter 28), you are perfectly positioned to participate.

### How to start:
1. **Find a Project:** Look for software you actually use (e.g., a VS Code extension, a Node library, or even documentation repositories like this one!).
2. **Look for "Good First Issue":** On GitHub, maintainers tag easy bugs with a label called `good first issue`. These are specifically reserved for beginners.
3. **Fork and Clone:** You click "Fork" to copy the repository to your account, then use `git clone` to pull it down to your Ubuntu SSD.
4. **Fix the Bug:** Use VS Code and the terminal to fix the code.
5. **Submit a Pull Request (PR):** You push the code back to GitHub and request that the maintainers merge it into the main project.

If your PR is accepted, your code is now running on thousands of computers worldwide, and it looks incredible on your resume.

## Tips & Warnings

> [!TIP]
> Your `README.md` is the most important file in any project. If a hiring manager visits your repository and there is no README explaining how to install and run the code, they will close the tab immediately. Always document your projects!

> [!WARNING]
> Don't get paralyzed by trying to build the "perfect" project. A small, simple project that is finished and deployed is worth 100x more than a massive, complex project that is only half-finished sitting on your hard drive.
